# 工具实现与安全机制深度分析

这份文档聚焦一条主线：

仓库里的 `read / write / bash / grep / glob` 这些高频工具，究竟是如何被注册、校验、授权、执行、回写结果，并在这个过程中落实安全约束与工程优化的。

它和《工具调用》以及《安全设计》的分工是：

- 《工具调用》更偏总体调用链与平台抽象
- 《安全设计》更偏权限体系与安全边界总览
- 本文档专门把几个关键工具按源码实现拆开，说明“具体怎么做”以及“为什么这样做”

---

## 1. 总览：从工具注册到结果回写的完整链路

这套工具系统不是“把几个函数暴露给模型”这么简单，而是一条分层流水线：

`Tool 定义 -> 工具池组装 -> 输入校验 -> Hook/权限判定 -> 工具执行 -> tool_result 序列化 -> transcript/UI 回写`

对应的关键代码位置是：

- `src/Tool.ts`
- `src/tools.ts`
- `src/services/tools/toolOrchestration.ts`
- `src/services/tools/toolExecution.ts`

### 1.1 `Tool` 抽象不是只有 `call()`

`src/Tool.ts` 里的 `Tool` 接口把一个工具拆成了多组能力：

- 结构层：`name`、`inputSchema`、`outputSchema`、`inputJSONSchema`
- 执行层：`call()`
- 安全层：`validateInput()`、`checkPermissions()`、`preparePermissionMatcher()`、`getPath()`
- 调度层：`isConcurrencySafe()`、`isReadOnly()`、`isDestructive()`
- 表达层：`prompt()`、`mapToolResultToToolResultBlockParam()`、多组 UI render 方法

这意味着工具不是孤立函数，而是一个跨越模型提示、客户端校验、权限系统、执行器、UI 和 telemetry 的平台对象。

### 1.2 `buildTool()` 的默认值体现了 fail-closed 思路

`buildTool()` 会为工具补默认实现：

- `isConcurrencySafe()` 默认 `false`
- `isReadOnly()` 默认 `false`
- `isDestructive()` 默认 `false`
- `checkPermissions()` 默认只是把判断交给通用权限系统

这里最关键的是前两项。默认“不并发”“不只读”，本质上是在未知情况下收紧能力，避免新工具因为遗漏配置而被错误地放入并发只读批次。

### 1.3 工具池先按权限过滤，再暴露给模型

`src/tools.ts` 里的 `getAllBaseTools()` 负责列出内建工具，`getTools()` 和 `filterToolsByDenyRules()` 再根据权限上下文、feature flag、REPL 模式、PowerShell 开关等做二次过滤。

这里的重点不是“调用时再拦”，而是“先缩小模型可见能力面”：

- 被 deny rule 整体命中的工具，会在暴露给模型前就从工具池移除
- 同样的逻辑也适用于 MCP 工具

这是一层很重要的防线：安全边界部分前移到了能力暴露阶段，而不只是运行时判定。

### 1.4 调度层对只读工具做并发批处理

`src/services/tools/toolOrchestration.ts` 会先调用 `partitionToolCalls()`，把一批 tool use 切成两类：

- 多个连续的并发安全工具，打成一个只读批次并发执行
- 其余工具串行执行

判断依据是工具自己的 `isConcurrencySafe()`。这就是为什么：

- `FileReadTool`、`GrepTool`、`GlobTool` 可以并发
- `FileWriteTool` 和大多数写工具会强制串行

这个设计不是为了“快一点”这么简单，更重要的是避免写写冲突、读写竞态和上下文修改顺序错乱。

### 1.5 执行层是一个多阶段管线

`src/services/tools/toolExecution.ts` 的主流程非常明确：

1. 用 Zod 做 schema 级输入解析
2. 调用工具自己的 `validateInput()`
3. 运行 `PreToolUse` hooks
4. 合并 hook 返回的 `updatedInput` / `hookPermissionResult`
5. 进入统一权限检查
6. 若允许，则执行 `tool.call()`
7. 把结果映射为 `tool_result`
8. 运行 `PostToolUse` hooks
9. 把最终结果写回 transcript / UI

这里值得注意的工程点有两个：

- `backfillObservableInput()` 只作用于“给观察者看的输入副本”，不直接污染最终 `call()` 的原始输入
- 如果某个 deferred tool 的 schema 没被发送给模型，`buildSchemaNotSentHint()` 会在输入校验失败时补充提示，告诉模型先通过 `ToolSearch` 加载 schema

## 2. `read` 工具：`FileReadTool` 的实现细节

核心文件：

- `src/tools/FileReadTool/FileReadTool.ts`
- `src/utils/readFileInRange.ts`

`FileReadTool` 是一个典型的“并发安全只读工具”：

- `isConcurrencySafe() -> true`
- `isReadOnly() -> true`

### 2.1 输入和输出不是单一文本模型

它的输入 schema 支持：

- `file_path`
- `offset`
- `limit`
- `pages`

输出 schema 不是单一字符串，而是判别联合：

- `text`
- `image`
- `notebook`
- `pdf`
- `parts`

这说明 `Read` 工具并不是“永远返回文本文件内容”，而是一个面向多种文件类型的统一读入口。

### 2.2 `backfillObservableInput()` 先做路径展开

它会把 `file_path` 做 `expandPath()`，原因不是为了读取方便，而是为了安全：

- hook 的 allowlist / rule matching 需要看见标准化后的绝对路径
- 否则 `~`、相对路径等写法可能绕过针对绝对路径的规则匹配

这是这套工具系统里一个非常常见的模式：给权限和 hook 看的路径一定尽量标准化。

### 2.3 `validateInput()` 主要做“轻量但关键”的前置检查

`FileReadTool.validateInput()` 不直接去做重读文件内容，而是先做一些低成本、高收益的判断：

- `pages` 字符串是否合法，范围是否超过 PDF 限额
- 路径经 `expandPath()` 后是否命中 read deny rule
- UNC 路径是否应该延后到授权后再做 I/O
- 扩展名是否是受支持的文本 / PDF / 图片
- 是否命中会卡死进程的设备文件

设备文件黑名单尤其值得注意，代码显式阻断了：

- `/dev/zero`
- `/dev/random`
- `/dev/urandom`
- `/dev/full`
- `/dev/stdin`
- `/dev/tty`
- `/dev/console`
- `/dev/stdout`
- `/dev/stderr`
- `/dev/fd/0-2`
- `/proc/.../fd/0-2`

这类检查的价值很高，因为它们不是传统意义上的“权限问题”，而是“工具可能永远卡住或输出无限流”的可靠性问题。

### 2.4 `checkPermissions()` 完全复用统一读权限层

`FileReadTool.checkPermissions()` 直接调用 `checkReadPermissionForTool()`。

也就是说，`Read` 自己不发明权限逻辑，而是把：

- 工作目录允许
- read deny / ask / allow rule
- “有 edit 权限则可 read”
- 内部路径豁免
- 工作区外提示和建议

都交给 `src/utils/permissions/filesystem.ts` 统一处理。

### 2.5 `call()` 不只是读文件，还带着多种工程策略

`FileReadTool.call()` 里有几层很值得注意：

- 读取时会拿默认的 `maxSizeBytes` 和 `maxTokens`，也允许调用方覆盖
- 会做同范围读取去重，如果文件的 mtime 没变，就返回 `FILE_UNCHANGED_STUB`
- 会根据路径触发技能目录发现和 conditional skills 激活
- 文件不存在时，会给出相似路径或 cwd 下建议路径

“同范围读取去重”是一个很典型的工程优化：

- 不是为了节省磁盘读
- 而是为了避免相同内容反复进入上下文，浪费 cache creation tokens

### 2.6 结果序列化按文件类型分流

`mapToolResultToToolResultBlockParam()` 会根据输出类型做不同映射：

- 图片转成 image content block
- notebook 转成 notebook cells
- PDF 返回元信息，并由其他路径补 document 内容
- 文本文件增加行号、freshness 信息和提醒信息
- 未变化则返回 stub

这里可以看出工具结果不是简单 `JSON.stringify(result.data)`，而是“模型侧表达”和“UI 侧表达”分离：

- 模型要拿到适合继续推理的内容
- UI 只显示摘要，不把全文都渲染出来

## 3. `write` 工具：`FileWriteTool` 的实现细节

核心文件：

- `src/tools/FileWriteTool/FileWriteTool.ts`

`FileWriteTool` 不是一个简单的覆盖写入器，它把“先读再写”“防并发覆盖”“编辑后通知周边系统”都放进去了。

### 3.1 输入非常简单，约束主要体现在流程上

输入只有两个字段：

- `file_path`
- `content`

约束不在 schema 复杂度，而在后面的验证与执行流程。

### 3.2 `backfillObservableInput()` 同样会标准化路径

这里和 `Read` 一样，都是为了确保 hooks 和权限匹配看的是展开后的绝对路径。

### 3.3 `validateInput()` 的核心目标是阻止“盲写”

`FileWriteTool.validateInput()` 做的最重要约束有：

- team memory 文件内容不能包含 secret
- 若 edit deny rule 命中，则直接拒绝
- UNC 路径不做额外 I/O，避免 NTLM credential leak
- 文件已存在时，必须先在 `readFileState` 里存在读取记录
- 若读取记录是 partial view，则禁止直接覆盖
- 若文件自上次读取后 mtime 已变化，则要求重新读取

这意味着 `Write` 不是“随便给路径和内容就写”，而是强制模型走一个更安全的编辑流程：

- 先读
- 再改
- 再写

从工程角度看，这直接降低了把用户未读过的文件整个覆盖掉的风险。

### 3.4 `checkPermissions()` 走统一写权限层

`FileWriteTool.checkPermissions()` 调用 `checkWritePermissionForTool()`，由统一文件系统权限层处理：

- edit deny / ask / allow rule
- internal editable path 豁免
- `.claude/**` 特殊 session 规则
- 危险路径阻断
- `acceptEdits` 模式下工作目录内自动放行

### 3.5 `call()` 在真正写盘前做了多轮保护

执行时有一条很明显的保护链：

1. 发现和激活相关 skill
2. `beforeFileEdited()` 做诊断跟踪前置处理
3. 先创建父目录
4. 可选做 file history 备份
5. 同步读取当前磁盘状态
6. 再次验证“上次读取后是否已变化”
7. 才真正写入磁盘

其中第 5-6 步尤其重要。它不是单纯依赖 earlier validate，而是在进入真正写盘前做最后一次原地确认，尽量收紧 TOCTOU 风险。

### 3.6 它会主动联动 LSP、diagnostics 和 VS Code

写完后它不是“结束了”，而是继续做系统联动：

- 清空已投递的 diagnostics
- 通知 LSP `didChange`
- 通知 LSP `didSave`
- 通知 VS Code 侧文件已更新

这说明文件工具不仅是改磁盘，还承担了“让 IDE 和语言服务状态保持一致”的职责。

### 3.7 行尾和编码策略是显式设计，不是默认行为

代码里有很明确的说明：

- 写入时不会去“继承旧文件的行尾风格”
- 而是把模型显式给出的内容当作权威输入
- 实际调用 `writeTextContent(..., 'LF')`

这是一种很明确的工程取舍：

- 避免历史文件 CRLF 把新内容 silently 污染
- 避免 repo 里二进制或其他文件影响新文件的换行策略

### 3.8 输出不仅有结果类型，还有结构化 patch

`Write` 的输出包含：

- `type: create | update`
- `filePath`
- `content`
- `structuredPatch`
- `originalFile`
- 可选 `gitDiff`

所以它既服务于“告诉模型写成功了”，也服务于 UI 和后续 diff 呈现。

## 4. `grep` / `glob`：只读搜索工具的实现细节

核心文件：

- `src/tools/GrepTool/GrepTool.ts`
- `src/tools/GlobTool/GlobTool.ts`
- `src/utils/glob.ts`
- `src/utils/ripgrep.ts`

这两个工具的共同点很明显：

- 都是只读
- 都可并发
- 都复用读权限系统
- 都以 `ripgrep` 为底层执行器

### 4.1 `GrepTool`：内容搜索工具

#### 输入层

它支持的参数非常丰富：

- `pattern`
- `path`
- `glob`
- `output_mode`
- `-A / -B / -C / context`
- `-n`
- `-i`
- `type`
- `head_limit`
- `offset`
- `multiline`

这说明它不是对 `rg` 的生硬代理，而是把最常用的一组能力做了受控映射。

#### 验证层

`validateInput()` 只验证：

- `path` 存在与否
- UNC 路径不在这个阶段做实际 I/O

#### 执行层

`call()` 会把参数映射成 ripgrep 命令行：

- 固定加入 `--hidden`
- 自动排除 `.git / .svn / .hg / .bzr / .jj / .sl`
- 用 `--max-columns 500` 防止超长行污染输出
- `multiline` 时才开启 `-U --multiline-dotall`
- `output_mode` 分别映射 `-l` 或 `-c`
- pattern 以 `-` 开头时，用 `-e` 防止被当成 flag
- `glob` 会做拆分，但保留 brace pattern
- 再叠加 ignore patterns 和 plugin cache exclusion

最后统一调用 `ripGrep(args, absolutePath, abortController.signal)`。

#### 输出层

它把结果分为三类：

- `content`
- `files_with_matches`
- `count`

并且在客户端自己做：

- `head_limit + offset` 分页
- 结果路径相对化，节省 token
- `appliedLimit / appliedOffset` 标注

这说明搜索工具并不是“把 rg 原样输出给模型”，而是做了一层受控整形。

### 4.2 `GlobTool`：文件枚举工具

#### 输入与验证

它的输入很简单：

- `pattern`
- `path`

`validateInput()` 只检查：

- path 是否存在
- path 是否为目录
- UNC 路径先不做 I/O

#### 权限与并发

它和 `GrepTool` 一样：

- `isConcurrencySafe() -> true`
- `isReadOnly() -> true`
- `checkPermissions()` 走 `checkReadPermissionForTool()`

#### 执行层

真正的实现落在 `src/utils/glob.ts`：

- 如果 pattern 是绝对路径，先用 `extractGlobBaseDirectory()` 拆出静态基目录和相对 pattern
- 再把 ignore patterns 归一化到目标搜索目录
- 实际执行是 `ripgrep --files --glob <pattern>`

这里最关键的一点是：它并没有自己在 Node 里递归遍历目录，而是继续复用 ripgrep。

这样做的工程收益很明确：

- 目录遍历性能更高
- 内存占用更可控
- ignore / hidden / glob 行为更统一

### 4.3 `ripgrep` 封装的工程细节

`src/utils/ripgrep.ts` 体现了很多“不是功能需求，但对工程质量很关键”的细节：

- 支持 system ripgrep / bundled ripgrep / embedded ripgrep 三种模式
- 避免直接执行绝对 `systemPath`，降低 PATH hijack 风险
- 设置较大的 `MAX_BUFFER_SIZE`
- 处理 `EAGAIN` 类资源不足错误
- 引入 `RipgrepTimeoutError`
- 对超时场景做 `SIGTERM -> SIGKILL` 升级
- 提供 `ripGrepStream()` 和 `ripGrepFileCount()`，避免所有场景都一次性把 stdout 全量缓冲进内存

这类封装说明搜索工具的真实复杂度并不在表面 schema，而在于“如何把一个高性能命令行工具安全地嵌入到 agent runtime 里”。

## 5. `bash` 工具：高风险工具的多层防线

核心文件：

- `src/tools/BashTool/BashTool.tsx`
- `src/tools/BashTool/bashPermissions.ts`
- `src/tools/BashTool/shouldUseSandbox.ts`

`BashTool` 是整套系统里实现和安全逻辑最厚的一类工具，因为它天然具备代码执行能力。

### 5.1 输入 schema 就已经体现了安全意图

它的输入不是只要 `command`：

- `command`
- `timeout`
- `description`
- `run_in_background`
- `dangerouslyDisableSandbox`

另外还有一个内部字段 `_simulatedSedEdit`，但模型侧 schema 会显式把它隐藏掉。

这里的设计非常重要：

- `_simulatedSedEdit` 只能由权限系统或预览流程注入
- 不能暴露给模型，否则模型可以绕过 sed 预览与权限检查，直接把它当成任意文件写入口

这是很典型的“内部控制通道和模型公开参数面分离”。

### 5.2 `isReadOnly()` 不是字符串猜测，而是语义检查

`BashTool.isReadOnly()` 会先看命令里是否包含 `cd`，再调用 `checkReadOnlyConstraints()` 做更系统的判断。

也就是说，shell 命令是否只读，不是简单看有没有 `cat/ls` 这些字样，而是有一套专门的 read-only 语义校验层。

### 5.3 `preparePermissionMatcher()` 为复合命令做了专门处理

它会解析命令结构，把 compound command 拆成多个 subcommand，再让 hook 规则对每个 subcommand 逐个匹配。

这样能避免类似：

- `ls && git push`

只因为整串字符串没直接匹配 `git *`，就绕过相关 hook。

### 5.4 `validateInput()` 主要阻止“错误的使用方式”

`BashTool.validateInput()` 当前重点阻断的是 sleep 模式：

- 如果 `MONITOR_TOOL` 开启
- 又没有请求后台运行
- 命令是长时间 `sleep`

就直接拒绝，要求：

- 阻塞型任务用 `run_in_background`
- 流式监控类场景用 `Monitor`

这不是传统的安全校验，而是“把 bash 从错误的工程使用方式里拉回来”。

### 5.5 `checkPermissions()` 是 bash 专属权限系统的入口

它调用的是 `bashToolHasPermission()`，而不是统一文件权限层。

`bashPermissions.ts` 里叠加了很多 bash 专属逻辑：

- 命令 AST 解析与语义分类
- subcommand 拆分与逐段判断
- exact / prefix / wildcard rule 匹配
- heredoc 与 multiline 命令的 rule suggestion 策略
- safe env var stripping
- safe wrapper stripping
- compound command 的 operator 权限检查
- classifier 接入
- sandbox 与 path/sed 约束联动

### 5.6 规则匹配不是简单 startsWith

`bashPermissions.ts` 里有几组很关键的辅助逻辑：

- `getSimpleCommandPrefix()`
- `getFirstWordPrefix()`
- `extractPrefixBeforeHeredoc()`
- `suggestionForExactCommand()`

这些代码的目的，是让权限规则尽量稳定、可复用、不过宽：

- 多行命令不保存成一整段 exact rule
- heredoc 优先提取稳定前缀
- 裸 shell 前缀如 `bash:*`、`sh:*`、`powershell:*` 明确列入危险前缀集合

这背后的工程考虑是：一条看似方便的宽规则，很容易退化成“事实上的任意代码执行放行”。

### 5.7 `shouldUseSandbox()` 明确把 sandbox 作为执行层决策

`src/tools/BashTool/shouldUseSandbox.ts` 的判断顺序是：

- 如果 sandbox 总开关没开，直接不用
- 如果显式要求禁用 sandbox，且策略允许，则不使用
- 如果命令为空，不使用
- 如果命中用户或动态配置里的 excluded commands，不使用
- 否则使用 sandbox

代码里还明确写了一个很重要的注释：

- `excludedCommands` 是用户便利特性，不是安全边界

真正的安全边界仍然是权限系统与 sandbox 本身，而不是“某条命令被排除在 sandbox 外”。

### 5.8 `call()` 不是一把 `exec()`，而是一整套 shell 运行时

`BashTool.call()` 的关键行为包括：

- 若存在 `_simulatedSedEdit`，直接走 `applySedEdit()`，不真的调用 sed
- 否则调用 `runShellCommand()`
- 收集 stdout/stderr、进度、后台任务信息
- 解释特殊返回码含义
- 把大输出复制到 tool-results 目录，必要时截断
- 检测图片输出并进行压缩或降采样
- 解析 CLI 发出的 Claude Code hint

也就是说，bash 执行路径里已经有了“小型 shell runtime + 结果管理器”的意味。

### 5.9 后台任务和长命令治理是 BashTool 的重要工程特性

它支持三类后台化路径：

- 模型显式设置 `run_in_background`
- 超时后自动后台化
- assistant mode 超过 blocking budget 后自动后台化

原因很现实：

- agent 不能长期被一个阻塞命令卡死
- UI 也不能因为一个长任务失去响应

所以 `BashTool` 不只是“运行命令”，还承担了把长命令纳入任务系统的职责。

## 6. 统一权限与路径安全层

核心文件：

- `src/utils/permissions/permissions.ts`
- `src/utils/permissions/filesystem.ts`
- `src/utils/permissions/pathValidation.ts`

### 6.1 通用权限系统先做工具级裁决

`permissions.ts` 的职责是：

- 统一读取 allow / deny / ask 规则
- 判断整个工具是否被规则整体命中
- 决定是否需要转交工具自己的 `checkPermissions()`
- 在非交互场景、hook 场景、classifier 场景统一记录 decision reason

它和工具专属权限逻辑的关系是：

- 先有通用框架
- 再允许工具追加自己的内容级判断

### 6.2 `checkReadPermissionForTool()` 的顺序体现了显式规则优先

`filesystem.ts` 里读权限的大致顺序是：

1. 提前阻断 UNC 和可疑 Windows 路径
2. 读 deny rule
3. 读 ask rule
4. 若已有 edit 权限，则可 read
5. 工作目录内自动允许
6. 内部可读路径允许
7. 读 allow rule
8. 否则 ask

有两个关键点：

- read deny / ask rule 优先级高于“有 edit 权限则可 read”
- 工作目录允许是在显式规则之后才生效的

这能保证用户显式写下的安全意图不会被默认便利行为覆盖。

### 6.3 `checkWritePermissionForTool()` 的顺序体现了写入更危险

写权限顺序更严格：

1. edit deny rule
2. internal editable path 豁免
3. `.claude/**` 的 session 级特殊 allow
4. `checkPathSafetyForAutoEdit()`
5. edit ask rule
6. `acceptEdits + working dir` 自动允许
7. edit allow rule
8. 否则 ask

这里最关键的是第 4 步：安全检查在普通 allow rule 之前。

换句话说，危险路径不是“只要用户写了 allow 就放行”，而是优先被安全检查拦住，至少要求更显式的人类确认。

### 6.4 `checkPathSafetyForAutoEdit()` 是路径级防逃逸核心

它会阻断很多看似边缘、但工程上很重要的危险目标：

- `.git`
- `.vscode`
- `.idea`
- `.claude`
- `.gitconfig`
- `.gitmodules`
- shell rc 文件
- Claude 自身 settings / commands / agents / skills 等目录
- UNC 路径
- 一系列可疑 Windows 路径模式

这些规则本质上是在保护两类边界：

- 能影响代码执行与环境初始化的配置文件
- 能影响 Claude 自身权限、命令、代理行为的元配置

### 6.5 `pathValidation.ts` 把很多路径绕过点直接做成显式检查

这一层有几个很典型的安全细节：

- `expandTilde()` 只允许普通 `~`，拒绝 `~user`、`~+`、`~-`
- 含 `$`、`%`、以 `=` 开头的 shell expansion 路径会被拒绝
- 写操作和创建操作禁止 glob pattern
- 读操作遇到 glob，只验证 glob 的 base directory
- 会对 canonical path 和 symlink 路径都做对称检查

这些都在防同一类问题：

- 我们校验的是一个字面路径
- 但 shell 或文件系统最终访问的是另一个展开后的真实路径

这正是典型 TOCTOU 与 path confusion 问题的来源。

### 6.6 sandbox 写白名单被当作额外允许路径

`pathValidation.ts` 还会把 sandbox 的 write allowlist 视为额外可写目录，但前提是：

- 路径已通过安全检查
- 不在 denyWithinAllow 内
- 不依赖它绕过工作目录内的 `acceptEdits` 约束

这个处理很务实：

- 让 `/tmp/claude/...` 这类已明确允许的路径少弹一次权限
- 但又不把 sandbox 白名单错误扩大成“所有工作区写操作都可绕过编辑模式”

## 7. 工程取舍与设计考虑

这几类工具和底层基础设施，清晰体现了几条工程原则。

### 7.1 只读工具并发，写工具串行

这是 `toolOrchestration.ts` 的核心设计之一。

收益是：

- 搜索和读取能明显提速
- 写操作保持顺序，避免上下文和磁盘状态错乱

### 7.2 默认收紧，而不是默认放开

从 `buildTool()` 的默认值到写操作禁止 glob，再到 path expansion 要求人工确认，整套设计都偏 fail-closed。

### 7.3 能在能力暴露阶段解决的问题，不拖到执行阶段

工具整体 deny rule 会直接把工具从工具池拿掉，不让模型先“看到再试着调用”。

### 7.4 路径安全是“原路径 + 解析路径 + 平台特性”三件事一起看

代码里反复强调：

- symlink
- canonical path
- UNC
- Windows ADS / 短文件名 / 长路径前缀 / trailing dot
- `/tmp` 与 `/private/tmp`

这说明路径安全在这里不是单点规则，而是跨平台的一组组合约束。

### 7.5 搜索和大输出都优先考虑内存与上下文成本

几个很典型的例子：

- `glob` 不用 Node 递归，复用 ripgrep
- `ripGrepStream()` 避免全量缓冲
- `Read` 的同范围去重降低上下文成本
- `Bash` 大输出落盘，并只给模型一个 preview + 路径

这些都是“Agent 运行时”和普通 CLI 工具之间很大的差别：它不仅要跑通命令，还要控制 token、上下文体积和会话流畅度。

### 7.6 把“不安全的便利功能”写成代码注释和结构约束

例如：

- `excludedCommands` 被明确标注不是安全边界
- `_simulatedSedEdit` 明确不进入模型公开 schema
- 宽泛 shell prefix 被明确禁止建议

这类写法的价值很高，因为它能让后续维护者知道哪些地方“看上去像优化，实际上不能随便改宽”。

## 9. 结论

如果只看表面，这几个工具似乎只是：

- 读文件
- 写文件
- 跑命令
- 搜索文本
- 枚举文件

但从源码实现看，它们背后其实共享了一套相当完整的 agent 工程基础设施：

- 统一工具抽象
- 按权限裁剪后的工具池
- 串并行调度
- hook 与权限系统
- 文件与路径安全
- shell 语义分析与 sandbox
- 大结果落盘与上下文节流

因此，真正值得关注的不是某个工具单点“能不能用”，而是这套系统如何在模型能力、用户权限、文件系统、shell 运行时和上下文成本之间维持平衡。

这也是这些实现细节最有价值的地方。
